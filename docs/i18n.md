## Architecture

`lib/I18n.php` exposes a single global `t($key, $params = [])` helper. Lookup chain:

1. URL `?lang=<code>` — sticky (writes a cookie).
2. Cookie `cleanmenu_locale`.
3. `users.locale` for authenticated users.
4. Tenant `default_locale` setting.
5. `Accept-Language` header best match.
6. Hardcoded fallback `ru`.

Locale files live under `locales/<code>.json`. Keys are dot-notated (`nav.menu`, `account.tab_security`, `errors.csrf_mismatch`). Parameter interpolation uses `{name}` placeholders — call as `t('greeting.hello', ['name' => $user['name']])`.

Missing keys fall back to the key itself in dev mode and the `ru` value in prod (so a missing `kk` translation degrades to Russian rather than showing `nav.menu`).

## Adding a new key

1. Add to `locales/ru.json` (the canonical reference).
2. Add to `locales/en.json` and `locales/kk.json` — every `ru` key must have an `en` and `kk` counterpart, even if the value is identical (the schema validator that's planned for Phase 9 will enforce this).
3. Replace the hardcoded string in PHP/JS:
   ```php
   <h2><?= htmlspecialchars(t('account.title')) ?></h2>
   ```
4. Re-run `php scripts/i18n-extract-strings.php` to confirm the line dropped from the candidate list.

## Surface migration tooling

`scripts/i18n-extract-strings.php` walks the project, flags lines with hardcoded Cyrillic that look user-visible, prints the top-30 most-repeated strings + top-20 most-affected files. JSON output via `--json --out=tmp/i18n-todo.json` for tooling.

```bash
php scripts/i18n-extract-strings.php
php scripts/i18n-extract-strings.php --min-len=8           # narrower, fewer candidates
php scripts/i18n-extract-strings.php --json --out=todo.json
```

Skipped paths: `vendor/`, `node_modules/`, `tests/visual/__snapshots__/`, `data/`, `.min.js`, CSS, images, audit-screens, the scanner itself.

False-positive minimisers:
- Skip lines that already use `t(` (assume migrated).
- Skip lines containing SQL keywords (INSERT/SELECT/UPDATE/DELETE/CREATE/ALTER) so query strings don't pollute the report.
- Require ≥ `--min-len` (default 4) contiguous Cyrillic characters.

## Known Gaps / Future Work

- **Surface migration is incomplete.** The scanner reports ~hundreds of remaining hardcoded RU strings across cart/account/owner/admin. Each migration is mechanical but additive — recommend chunked commits per surface.
- **No locale validator.** A CI guard that fails if `en.json` or `kk.json` is missing keys present in `ru.json` is planned for Phase 9 (compliance pack).
- **No JS-side i18n.** Frontend interactive strings (alerts in cart/order flow) currently live in `js/cart.min.js` etc. as RU literals. A small `window.i18n.t()` shim that reads from a script-injected JSON dict is planned.
- **No RTL support.** Not relevant for ru/en/kk but tracked for the eventual ar/he expansion.
