## 0. Краткое резюме

Menu Labus сегодня — это **зрелый production-grade white-label движок меню и заказов ресторана**, значительно больше, чем «простое онлайн-меню». В коде уже есть разделение провайдер/тенант на уровне рантайма, отдельные БД на каждого тенанта, REST API v1 для мобильного приложения, PWA-оболочка с push-уведомлениями, аналитика для владельца, бэкофис для персонала/администратора, уведомления через Telegram-бота, интеграции YooKassa + T-Bank, поддержка СБП, модификаторы, чаевые, ABC-анализ, QR-меню на столах, онбординг-мастер, и полностью автоматизированный git-based деплой с pre-push проверками документации/OpenAPI/mojibake и post-merge браузерной регрессией.

Документация структурирована и в основном честна, но **заметно отстаёт от кода**. Последний полный цикл аудита пришёлся на `2026-03-23 / 2026-03-25 / 2026-03-26`, и почти каждый документ несёт эту метку. С тех пор код получил как минимум: редизайн hero-экрана (сегодня), полировку UI модификаторов, CSRF-fallback в админских модулях, переупорядочивание админского каталога, перевод табличных действий в столбик на планшетных ширинах, версионирование asset-а admin-modifiers. Только UX/UI-роадмап отражает последний кусок изменений, остальные документы не обновлялись.

Более крупный документационный пробел — это **целые функциональные области, существующие в коде, но не описанные нигде**: платёжный слой (YooKassa + T-Bank + СБП + наличные), Telegram-бот, модификаторы, чаевые, PWA push, ABC-анализ, Telegram-алерты по стоп-листу, онбординг-мастер, шаблонизация nginx для кастомных доменов и вся директория `sql/` миграций. Некоторые из них упомянуты косвенно (в feature matrix стоит «implemented»), но нет документа, который оператор или новый инженер мог бы прочитать, чтобы понять механику.

С точки зрения конкуренции продукт ближе к **Zenky / Restik / Foodeon** (российские SaaS-конструкторы сайтов меню с доставкой + интеграциями POS), чем к iiko / r_keeper / Toast (полные POS-экосистемы). Чтобы выйти в топ-1 своего реального сегмента — «кастомизированный клиентский сайт ресторана с онлайн-заказом и лёгким операционным слоем» — у Menu Labus уже есть большинство кирпичиков. Чего **пока нет**: глубокого CRM и сегментации, автоматизированного маркетинга (email/SMS/push-сценарии), отзывов и репутации, броней столов, контракта KDS (кухонного дисплея), адаптеров iiko/r_keeper, AI-слоя (рекомендации, прогноз спроса, сентимент-анализ отзывов, апселл-подсказки). Закрытие этих пунктов плюс небольшая инвестиция в QA — самый короткий путь к доминированию.

Документ разбит на:

1. **Раздел A** — документация vs. реализация: конкретные несоответствия, противоречия, расхождение.
2. **Раздел B** — функциональные области, которые живут в коде, но (почти) не документированы.
3. **Раздел C** — устаревшие документы, с рекомендацией по каждому файлу.
4. **Раздел D** — конкурентный бенчмарк: мировой топ-10 + российский топ-3, сравнение с Menu Labus.
5. **Раздел E** — стратегический роадмап выхода в топ-1 сегмента.
6. **Раздел F** — рекомендуемый порядок ближайших действий (что делать со следующего понедельника).

Оба утверждения противоречат более свежим документам:

И код это подтверждает: `scripts/tenant/go-live.sh`, `lib/tenant/launch-contract.php`, обработка `public_entry_mode` в `session_init.php` + `api/save/brand.php` + `index.php` — всё существует.

**Рекомендация:** переписать `README.md` одним проходом, синхронизировав с feature matrix. Это первый файл, который видит новый инженер или потенциальный клиент, и сейчас он занижает продукт.

### A.2. Расхождение UX/UI-роадмапа (минорное, уже закрывается)

### A.3. Docs-drift guard срабатывает на чисто визуальных изменениях hero

`scripts/docs/check-doc-drift.sh` классифицирует `index.php` и `css/*` как контрактные поверхности `launch` / `shell`. Сегодня это сработало на чисто визуальном редизайне, заставив обновить документ одной строкой. Страж работает как задумано, но эвристика шире, чем «контракт». Два варианта:

- **Оставить как есть** — небольшой налог, стимулирует гигиену документации.
- **Сузить классификатор** — исключить `css/*` из поверхности `shell`, или требовать, чтобы изменения `.php` задевали бизнес-логику, а не только разметку.

**Рекомендация:** оставить страж, но вынести определение «контракта» в короткую шапку `check-doc-drift.sh` и в `docs/dev/git-hooks.md`, и подумать о переводе `css/*` из блокирующего уровня в предупреждение. Несрочно.

### A.4. Security hardening roadmap застыл на Phase 2/3/5 = Partial

`docs/security-hardening-roadmap.md` остаётся `Partial` для Phase 2 (сетевая защита), Phase 3 (SSH/fail2ban) и Phase 5 (ежемесячный ритм проверки) с `2026-03-23`. Скрипты для всех трёх фаз (`apply-network-policy.sh`, `harden-ssh-fail2ban.sh`, `monthly-review.sh`) есть в репозитории. Чего не хватает — **свидетельств выполнения на боевом хосте**: нет ни одной записи в change-log (`docs/security-change-log-template.md` до сих пор пустой шаблон), нет зафиксированного allowlist, нет подтверждения fail2ban.

Тип пробела: **операционное расхождение, не код.** Код готов, операционный ритм его не подхватил.

**Рекомендация:** провести одну фазу безопасности от начала до конца в ближайшее тихое окно (Phase 2 — самая безопасная точка старта), зафиксировать результат в `docs/security-change-log-<дата>.md` (переименовать шаблон), обновить статус фазы на `Implemented`. Остальные фазы пойдут по той же схеме.

### A.7. `docs/api-smoke.md` корректен, защита smoke-раннера — nginx + PHP guard

`scripts/api-smoke-runner.php` корректно отказывает в веб-доступе (guard `PHP_SAPI !== 'cli'` + 404). Блокировка на уровне nginx тоже есть в `nginx-optimized.conf` и `deploy/nginx/custom-domain-template.conf` (`location ^~ /scripts/ { return 404; }`). Оба слоя на месте. Хороший пример согласованности документации, скрипта и конфигурации nginx. **Действий не требуется.**

### A.8. OpenAPI покрывает только часть реальной API-поверхности

`docs/openapi.yaml` — авторитетный контракт мобильного API, pre-push hook его проверяет для пушей в `main`. Но многие эндпоинты, которые приложение реально использует в браузере (CSRF-защищённые POST из бэкофиса и пользовательских flow), **не** замоделированы в OpenAPI, потому что OpenAPI ограничен `api/v1/*`. Решение об охвате легитимное, но тогда частая фраза в документации «API contract source of truth: `docs/openapi.yaml`» слегка вводит в заблуждение: это **контракт мобильного API**, а не API в целом. У веб-эндпоинтов (`api/save/brand.php`, `api/save/colors.php`, `toggle-available.php`, `update_order_status.php`, `update_user_role.php`, `api/checkout/cash-payment.php`, `generate-payment-link.php` и т. п.) **нет схемы**.

**Рекомендация:** либо (a) переименовать файл OpenAPI в `mobile-api.yaml` и явно это прописать, либо (b) добавить второй, более тонкий `docs/web-endpoints.yaml`, перечисляющий веб-POST, их CSRF-ожидания и формат ответов. Вариант (a) — дешёвый фикс; вариант (b) окупается при QA и внешних аудитах.

## Раздел D — Конкурентный бенчмарк

### D.1. С кем реально конкурирует Menu Labus

Menu Labus — **не POS**. Он не владеет кассовым ящиком, KDS, складским учётом, расчётом зарплаты и ингредиентным костингом. Чем он **владеет** — клиентским сайтом, онлайн-заказом, гостевой PWA, тенант-брендированным мобильным API, лёгким бэкофисом сотрудника/админа/владельца и операторским инструментарием запуска тенанта.

Честная группа пиров:

- **Россия:** Zenky, Restik, Foodeon. (SaaS-конструкторы сайтов меню с онлайн-заказом и адаптерами POS.)
- **Мир:** ChowNow, Flipdish, BentoBox, Popmenu, UpMenu, Slice, Lunchbox. (Сайт + прямой заказ + маркетинговый слой.)

Гиганты (iiko, r_keeper, Toast, Square for Restaurants, Lightspeed, Clover, TouchBistro) — **смежные**, не прямые конкуренты. Menu Labus не должен пытаться стать iiko; он должен **интегрироваться** с iiko так же, как это делает Zenky, и владеть клиентским слоем и запуском тенанта, где iiko слаб.

### D.4. Структурные преимущества, которые у Menu Labus уже есть

По результатам обхода кода Menu Labus **сильнее большинства пиров** в следующем:

1. **Жёсткая изоляция данных.** «Один тенант = одна БД» редкость в этом сегменте. Zenky/Restik/Foodeon — мульти-тенант в одной БД. Для чувствительных к приватности клиентов (сети, франшизы, премиум-рестораны) это настоящий дифференциатор.
2. **Строгий CSP + no-inline.** Большинство конструкторов сайтов ресторанов везут инлайн-стили и eval. У Menu Labus работает nonce-based CSP — это энтерпрайз-безопасность.
3. **Реальная автоматизация запуска тенанта** (`scripts/tenant/go-live.sh`, launch-артефакты, smoke, security smoke, браузерная регрессия) — у большинства конкурентов эквивалента нет, онбординг тенанта ручной.
4. **Серверно рендеримое меню PHP, хорошо для SEO.** Текст меню — это серверный HTML, не JS-hydrated. Реальное преимущество перед Zenky.
5. **Глубина платёжных провайдеров под РФ**: YooKassa + T-Bank + СБП + наличные — best-in-class для российского рынка. Большинство глобальных платформ СБП вообще не поддерживают.
6. **Git-based деплой с стражами docs-drift + OpenAPI + mojibake.** Редкость даже на уровне iiko/Toast.

### D.5. Структурные пробелы, которые есть у Menu Labus сегодня

1. **Нет модуля броней столов.** Table-stakes для BentoBox, Popmenu, Toast, Lightspeed. У Menu Labus есть «Оставить заявку», но это контактная форма, а не бронь стола с посадкой и слотом.
2. **Нет email / SMS маркетингового слоя.** Push есть, но сегментации акций нет.
3. **Нет модуля отзывов / репутации.** Ни сбора отзывов, ни flow к Google Reviews, ни сентимент-поверхности для владельца.
4. **Нет лояльности / баллов / рефералов.** Базовый повтор заказа есть, но нет баллов, уровней, рефералов.
5. **Нет адаптера iiko / r_keeper / Frontpad.** Единственный крупнейший пробел для расширения на российский рынок — гиганты владеют кухней, Menu Labus не может сейчас продаваться туда, где уже стоит iiko.
6. **Нет KDS (кухонного дисплея).** Очередь сотрудника — браузерная, не tablet-first.
7. **Нет AI-слоя.** Ни рекомендаций, ни прогноза спроса, ни сентимент-анализа, ни авто-описаний блюд, ни апселл-подсказок, ни чатбота броней. Конкуренты 2026 года считают это table-stakes.
8. **Нет in-product биллинга / self-serve онбординга.** Новым тенантам нужен оператор, запускающий `launch.php`. Конкуренты дают владельцу ресторана самому подписаться и заплатить.
9. **Нет per-tenant сборки мобильного приложения.** Capacitor-враппер прибит к провайдеру.
10. **Нет юнит-тестов** (см. B.10). Это пробел **девелоперской скорости**, а не видимый пользователю, но он ограничивает скорость, с которой можно безопасно добавлять все перечисленное выше.

### E.2. Ближняя перспектива (недели 3–6): закрыть продуктовые table-stakes

1. **Модуль броней.** Стол + количество гостей + слот времени + контакты гостя. Минимальная версия: ключ настройки на тенанта (включить/выключить), одна таблица БД, один публичный виджет на тенант-домашней, одна страница владельца. Базовый уровень BentoBox.
2. **Отзывы / feedback loop.** После завершения заказа — один шаг обратной связи в PWA (1–5 звёзд + текст), поверхность владельца с последними 50, глубокая ссылка «опубликовать в Google» для 5-звёздочных.
3. **Лояльность v1.** Модель «1₽ = 1 балл». Списание на чекауте. Одна таблица, один переключатель, один отчёт. Достаточно, чтобы иметь значение, не превращаясь в полноценную платформу лояльности.
4. **Email/SMS-кампании v1.** Одна кнопка «рассылка всем клиентам» в админе с простым шаблоном. Не строить сегментацию сразу. Использовать существующий `mailer.php` и добавить SMS-адаптер (один провайдер, например SMS.ru или МТС Exolve).
5. **Документы броней/отзывов/лояльности.** Каждый — с собственным `docs/*.md`.

### E.3. Средняя перспектива (недели 7–12): интеграции, разблокирующие РФ-рынок

1. **Адаптер iiko.** Читать меню из iiko, отправлять заказы в iiko, зеркалить доступность позиций, синхронизировать стоп-лист. Эта интеграция открывает двери каждому ресторану, где уже стоит iiko — а таких тысячи.
2. **Адаптер Frontpad.** То же, но для бюджетного сегмента. Быстрее интегрируется, чем iiko.
3. **Адаптер r_keeper.** Legacy, но всё ещё большая установленная база. Ниже приоритета, чем iiko/Frontpad.
4. **Проброс Yandex.Еда / Delivery Club.** Большинство ресторанов с iiko уже ходят в агрегаторы через iiko. Если Menu Labus стоит впереди, ему нужно просто не сломать поток. Явно задокументировать.
5. **Per-tenant сборка Capacitor.** Небольшой CLI (`scripts/mobile/build-tenant-app.sh`) — на вход слаг тенанта, на выход Android APK + iOS архив с подставленным брендингом. Без автоматизации App Store в v1, но иметь артефакт — половина дела.
6. **KDS v1.** Второй URL, который кухонный планшет «прикрепляет» — те же данные, что и очередь сотрудника, но переформатированные под дистанцию чтения на кухне, со звуком на новый заказ и авто-переходом по смене статуса.

### E.4. Средне-дальняя перспектива (месяцы 4–6): AI-слой и удержание

1. **AI-генератор описаний блюд.** Админ нажимает кнопку, видит три варианта описания (короткое, среднее, «аппетитное»), выбирает один. Использует Claude или локальную LLM. Огромная экономия времени при онбординге каталога нового тенанта.
2. **Рекомендательный движок v1.** «Клиенты, заказавшие X, также брали Y» на карточке товара. Простой co-occurrence запрос по `order_items`. Начинает работать после нескольких сотен заказов, улучшается естественно.
3. **Прогноз спроса для владельца.** По последним 90 дням заказов + день недели + фид погоды предсказывать ожидаемое число заказов на завтра по категориям. Владельцы это любят. Дёшево на линейной модели.
4. **Сентимент отзывов для владельца.** Кластеризовать последние 100 текстовых отзывов по сентименту и теме, вывести на дашборд владельца.
5. **Апселл-подсказки на чекауте.** «Добавить картошку за 150₽?» — правила в первую очередь, ML во вторую.

### E.5. Постоянно: инженерная гигиена и скорость разработки
